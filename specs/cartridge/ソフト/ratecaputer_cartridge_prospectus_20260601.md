**目論見書 — 2026年6月**
**対象ハードウェア: CAR-01（UIAPduino版）**
**MCU: CH32V003 RISC-V 48MHz**
**策定: 2026年5月〜6月**
**ステータス: 設計検討中**

---

## 1. 背景と目的

ラテカピューターカートリッジ（CAR-01）の共通プログラムは実装・サイズ検証がほぼ完了し（共通5,320B・UIAPduino込み7,368B・余裕312B）、実機試験に向けた段階に入った。次の課題は実用アプリを動作させるための開発環境一式の整備である。

本書はその整備計画のうち「実機試験で実用アプリの動作検証が可能になる最初のステップ」に必要な作業を整理し、優先順位・方針・設計上の論点をまとめたものである。

## 3. RcApi と Arduboy2 のギャップ分析

RcApiは共通プログラムの固定アドレスAPIテーブル（12関数）として完成しており、システム制御・入出力の基盤としての完成度は高い。Arduboy2との比較では「描画系」にギャップが集中するが、これはRcApiを壊して互換化するのではなく、2層ライブラリとしてSPI Flash上に別途提供する方針とする。

### 現在の RcApi 12関数

**ディスプレイ非依存（機種共通）:**

```
iap_run()            アプリ切り替え（パターンB・戻らない）
iap_call()           モジュール呼び出し（パターンA・戻る）
iap_return()         呼び出し元に戻る
load_resource()      SPI Flash→FRAM作業領域へリソースロード
keyscan_get()        ボタン状態取得（ビットマスク）
keyscan_wait()       ボタン押下待機
keyscan_get_button() 特定ボタン確認
```

**ディスプレイ依存（機種ごとに実装が異なる）:**

```
disp_init()          初期化
disp_fill()          全面塗りつぶし
disp_draw_string()   文字列描画（恵梨沙フォント・位置指定込み）
disp_set_window()    描画ウィンドウ指定
disp_write_pixel()   1ピクセル書き込み（→2層移管を検討）
```

### ギャップ対比表

| カテゴリ | Arduboy2 | RcApi現状 | 方針 |
|---------|---------|----------|------|
| 初期化・制御 | begin() boot() | disp_init() + boot.c | 1層 ✅ |
| テキスト描画 | drawChar() setCursor() | disp_draw_string()（恵梨沙・位置指定込み） | 1層 ✅ ／文字サイズ変更のみ2層 |
| 基本入力 | pressed() buttonsState() | keyscan_get() keyscan_get_button() | 1層 ✅ |
| justPressed / justReleased | pollButtons()系 | なし | 2層（フレーム制御と依存関係あり） |
| ピクセル・図形描画 | drawPixel() drawLine() drawRect()… | disp_write_pixel()のみ→2層へ移管 | 2層 |
| スプライト描画 | Sprites::drawOverwrite()他4種 | なし | 2層（移植の核心） |
| ビットマップ | drawBitmap() drawCompressed() | なし | 2層 |
| フレーム制御 | nextFrame() setFrameRate() | なし（概念自体なし） | 2層（justPressed等と依存） |
| 衝突判定・乱数 | collide() initRandomSeed() | なし | 2層 |
| サウンド | audio.on/off() BeepPin1::tone() | YMF825（3層） | 3層（設計上意図的に別扱い） |
| セーブデータ | EEPROM直接 | セーブデータAPI（2層・将来） | 2層（FRAM保護領域に限定） |
| リソース管理 | なし | load_resource() | 1層 ✅ RcApi独自の強み |
| モジュール切替 | なし | iap_run() iap_call() iap_return() | 1層 ✅ RcApi独自の強み |

### フレーム制御と入力の依存関係

```
フレーム制御（nextFrame）があって初めて
    ↓
「前フレーム」という時間単位が定義されて
    ↓
justPressed / justReleased が意味を持つ
```

このため justPressed / justReleased はフレーム制御とセットで2層ライブラリに持っていく。

## 4. e-ink フレームバッファ問題

### 現状の設計

現状のeink.cはTFTと同様に「直接転送」方式で実装されており、FRAMバッファを持たない。e-inkチップ（SSD1681）内部のRAMへSPIで転送し、eink_full_update()で一括更新する構造である。書いたデータをCH32V003側から読み返す手段がない。

### TFTとの違い

TFT（ST7789）は内蔵SRAM 2KBの制約上フレームバッファを持てないが、描画しながら直接転送できるため実用上問題は少ない。e-inkは更新が遅く（フル更新約2秒・部分更新約0.3秒）、部分更新のためには更新しない領域のデータも含む5,000バイト全体を再送する必要がある。

### 部分更新が必要なケース

| ケース | 内容 |
|--------|------|
| ポケピのアニメーション | 背景・ステータスは固定・キャラクタースプライト部分のみ差し替え |
| テキスト更新 | ノベルゲームのメッセージウィンドウのみ切り替え |
| UI要素の更新 | メニューカーソル位置・HPゲージ等 |
| 数値表示 | 時計・センサー値の数字部分のみ更新 |

### 必要な対応

FRAMのSRAM相当作業領域（最低192KB）に5,000バイト（200×200÷8）のフレームバッファを設ける。これはポケピ育成ゲームという主用途の実現に必須であり、早期に設計に組み込む必要がある。

## 5. ステップ1 実現計画

実機試験で実用アプリの動作検証が可能な状態を「ステップ1完了」と定義する。

### Phase 1（最優先）— 書き込み・読み出しツール群

**対象:** write_program.py / log_capture.py / dummy_data_gen.py

アダプタケーブルは不要——USB-C直結でUSB HID経由が完結する設計。ただし書き込みツール自体が未実装であり最優先。HIDレポートフォーマットは確定済み（64B固定・cmd/seq/addr/length/data[56]）。

**確認が必要な事項:**
- rv003usbのHIDディスクリプタがWindows環境でドライバ不要で動作するか（実機確認）

### Phase 1（並行）— テキスト変換ツール（char_convert.py）

UTF-8 / Shift-JIS文字列 → 恵梨沙フォントインデックス列（uint16_t[]）の双方向変換。TFT表示の最低限テストに必須。インデックス → 文字列の逆変換も含む。

### Phase 2 — TFT表示最低限テスト

Phase 1のツールでデータ生成 → write_program.pyで実機に書き込み → TFT上に文字列・図形が表示されることを確認。ここで書き込みツールの動作が検証される。

### Phase 3 — Arduboyゲーム移植

**移植方針:**

8KBのApp Areaにそのまま収まるArduboyゲームは存在しない（最小のpicovadersでも17KB）。Arduboy2ライブラリ部分（約8〜10KB）を2層ライブラリとしてSPI Flash共通モジュール領域に分離し、ゲームロジック単体（約7〜9KB）をApp Areaに収める方式で移植する。

移植コンバータは「Arduboy2のAPIコール → 2層ライブラリのAPIコール」に変換する形となる。

**移植対象ゲーム候補:**

| タイトル | ジャンル | ライセンス | バイナリサイズ | 優先度 |
|---------|---------|-----------|--------------|--------|
| picovaders | シューティング | MIT | 17.1 KB（実測） | ★★★ 最初 |
| Shadow Runner | アクション | MIT | 推定20KB台 | ★★☆ 次点 |
| Mystic Balloon | アクション | MIT | 推定20KB台 | ★★☆ |
| Arduventure | RPG | MIT | 推定28KB前後 | ★☆☆ 後工程 |
| Catacombs of the Damned | 擬似3D FPS | MIT | 32.5 KB（実測） | ★☆☆ 後工程 |
| MicroCity | シミュレーション | GPL-2 | 推定28KB前後 | ★☆☆ 参考 |

### Phase 4 — エミュレータ構築

**目的:** バグが環境起因かアプリ起因かを切り分ける。

**ベース:** Spike（riscv-isa-sim）を採用。

| 選択肢 | ライセンス | 採否 | 理由 |
|--------|-----------|------|------|
| Spike（riscv-isa-sim） | BSD | ✅ 採用 | 配布制約なし・RV32EC対応・軽量 |
| QEMU | GPL v2 | ⚠️ 条件付き | 配布時ソース公開義務・周辺実装例が豊富 |
| Wokwi | プロプライエタリ | ❌ 不採用 | 改変・組み込み不可 |
| Velxio | AGPL v3 | ❌ 不採用 | GPLより厳しい |

**実装範囲（最小構成）:**

| 対象 | 実装方法 |
|------|---------|
| CPUコア（RV32EC） | Spikeがそのまま対応 |
| 内蔵Flash・SRAM | Spikeのメモリマップに追加 |
| SPI Flash・FRAM | Spikeプラグイン（C++・数百行） |
| TFT（ST7789） | フレームバッファをPNG/SDL出力するスタブ |
| デバッガ | Spike標準のインタラクティブモード（レジスタダンプ・ステップ実行） |

周辺デバイスの完全シミュレートはステップ2以降。
