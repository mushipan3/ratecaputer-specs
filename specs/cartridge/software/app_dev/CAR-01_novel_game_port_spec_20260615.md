# ノベルゲーム移植目論見書

## ラテカピュータ CAR-01（UIAPduino/TFT版）対応

**対象ハードウェア**: CH32V003F4P6 (Flash: 16KB / SRAM: 2KB)  
**外部ストレージ**: SPI Flash (16MB), SPI FRAM (256KB/512KB)  
**出力デバイス**: ST7789 TFT液晶 (240×240px, 1.3inch)  
**音源ハード**: YMF825 (FM音源モジュール)  
**検証素材**: Fate/stay night (TYPE-MOON / 吉里吉里KAG形式)  
**Rev. 1.0 — 2026年5月**

-----

## 1. エグゼクティブ・サマリー

本目論見書は、既存の著名なスクリプト形式ノベルゲーム（吉里吉里/KAG、NScripter、Ren’Py等）の資産を、極小リソースのMCU環境「CAR-01」へ移植するための技術検証およびシステム設計書である。

検証素材として *Fate/stay night*（吉里吉里/KAG形式・元解像度800×600）を採用し、以下のアプローチの現実性を証明する。

- **PC側での静的解析・全自動コンパイル（トランスパイル）**: 既存スクリプトを1ブロック最大9KB以下のバイトコードバイナリへ機械的に変換
- **実機側での1ブロック完結型IAP駆動**: ソフトウェアリセット方式によるシームレスなブロック継続再生
- **変数の寿命別分離**: 長寿命フラグをFRAMに、短寿命フラグをSRAMに配置

これにより、数メガバイトに及ぶ巨大タイトルを、16KB Flash / 2KB SRAMの制約下で100%ネイティブ動作させる現実解を証明する。

-----

## 2. 物理ディスプレイ・レイアウト設計

ST7789（240×240px）の物理画面を、論理的・物理的に独立した2つの駆動領域に分断する。

### 2.1 領域定義

|領域            |Y座標範囲      |サイズ      |用途               |
|--------------|-----------|---------|-----------------|
|**ゲーム画面領域**   |Y=0〜Y=179  |240×180px|背景・立ち絵・イベントCG    |
|**メッセージ・UI領域**|Y=180〜Y=239|240×60px |セリフ・話者名・選択肢・システム行|

### 2.2 ゲーム画面領域（240×180px）

アスペクト比4:3。*Fate/stay night*の元解像度800×600と完全一致するため、画像の歪みなしに30%縮小で対応できる。

元解像度別の縮小後サイズと表示方法：

|元解像度     |アスペクト比|縮小後サイズ |表示方法                     |
|---------|------|-------|-------------------------|
|800×600  |4:3   |240×180|ゲーム画面全域にぴったり収まる          |
|640×480  |4:3   |240×180|同上                       |
|1920×1080|16:9  |240×135|上端寄せ。空き45pxはメッセージ・UI領域に吸収|
|1280×720 |16:9  |240×135|同上                       |

16:9等、ゲーム画面領域（180px高）に満たない縦サイズの場合は**上端寄せ**とし、余白分はメッセージ・UI領域の高さが広がる形で吸収する。上部に余白は生じない。

### 2.3 メッセージ・UI領域（240×60px）

黒背景固定。恵梨沙フォント（8×8px）を使用。

|項目   |仕様                          |
|-----|----------------------------|
|フォント |恵梨沙フォント 8×8px               |
|最大文字数|30文字×6〜7行（最大210文字）          |
|最下行  |システム・ステータス行として固定（「▼」待機アイコン等）|
|背景   |黒固定（アルファブレンド処理を完全排除）        |
|話者名表示|テキスト領域最上行に表示                |

### 2.4 描画・転送の最適化

**(1) 背景透過処理の全廃**

メッセージ・UI領域を黒背景固定とすることで、VRAM上でのアルファブレンドや透過合成を一切排除。1bitのフォントビットマップデータをそのまま液晶へ流し込む超軽量描画を実現する。

**(2) 矩形部分Flush（帯域節約）**

コマンド CASET / RASET を用い、更新単位を必要最小限に絞る。

|更新内容      |転送範囲           |
|----------|---------------|
|背景・CG切り替え |上部 240×180px のみ|
|テキスト1行進行  |下部の当該1行分のみ     |
|メッセージ画面全消去|下部 240×60px    |

24MHz SPIバスの転送オーバーヘッドを最小化する。

## 3. メモリおよびストレージ・アーキテクチャ

### 3.1 物理配置と役割定義

#### 内蔵Flash（16KB）

```
0x0000〜0x17FF（先頭6KB）: 共通プログラム領域
  TFT駆動・SPIマネージャ・IAP制御
0x1C00〜0x3FFF（残り9KB）: App Area（両機種統一）
  ノベルゲームインタプリタ本体 + オーディオ再生機能（〜1KB）
  IAPによりブロック単位で随時書き換わる
```

#### SPI FRAM（256KB/512KB）

```
0x00000〜0x01FFF（  8KB）: システム管理領域（SPI Flashカタログキャッシュ等）
0x02000〜0x05FFF（ 16KB）: セーブデータ領域（詳細は第4章）
0x06000〜0x07FFF（  8KB）: 共通プログラムバックアップ（自己復元用）
0x08000〜0x0FFFF（ 32KB）: SPI Flash管理処理
0x10000〜0x13800（〜14KB）: コンテキストスタック（最大7段）※汎用
0x13800〜0x21C00（ 55KB）: 恵梨沙フォント展開領域（起動時にSPI Flashから転送）
0x21C00〜       （残り）:  アプリ自由作業領域
  プレミアム版（512KB）: 約365KB
  スタンダード版（256KB）: 約109KB
```

**注**: ノベルゲームインタプリタはIAPブロック切り替えをコンテキストスイッチとして使わず、ソフトリセット方式で動作する（後述3.2節）。コンテキストスタック領域はこのアプリでは使用しない。

#### SPI Flash（16MB）

読み出し専用マスターデータストレージ。

```
0x000000 ┌──────────────────────┐
         │ カタログテーブル      │ 32KB固定
         │ 58エントリ × 538B    │ 起動時にFRAM管理領域（8KB）へキャッシュ
0x008000 ├──────────────────────┤
         │ 恵梨沙フォント        │ 64KB固定（6877文字×8B=55,016B）
         │ （固定アドレス）      │ ブートローダ起動時にFRAM作業領域へ展開
0x018000 ├──────────────────────┤
         │ システム予約領域      │ 512KB
0x098000 ├──────────────────────┤
         │ アプリ領域            │ 約15.4MB・4KBブロック×3968本
         │ 4KBブロック単位管理   │ 各アプリに1〜複数ブロックを割り当て
0xFFFFFF └──────────────────────┘
```

**カタログエントリ構造体（確定版）：**

```c
typedef struct {
    uint8_t  valid;              // 0xFF=空き / 0x01=有効 / 0x00=論理削除
    uint8_t  disp_type;          // 0x01=e-ink / 0x02=TFT・OLED（App Area 9KB・両機種統一）
    uint16_t title[8];           // 恵梨沙フォントインデックス列（最大8文字）
    uint32_t flash_addr;         // SPI Flash上の先頭アドレス（4KBアライン）
    uint32_t total_size;         // 全体サイズ（バイト）
    uint8_t  block_bitmap[512];  // 4096ブロック×1bit・重複検知用
} CatalogEntry;  // 計538バイト
```

**ブロック管理方針：**
各アプリのリソースは4KBブロック単位で管理する。リソースが複数ブロックにまたがる場合、ブロックが連続していることが前提。書き込み時にPC側ツールが連続空き領域を確保する（連続空きがない場合はデフラグ後に配置）。

### 3.1b アプリ内ブロックディレクトリ

プログラムブロック後半のメタ情報領域（6KB・TFT/OLED機種）に格納。PC側ビルドツールが自動生成。

**リソースエントリ構造体（8バイト固定）：**

```c
typedef struct {
    uint16_t block_cell;    // [15:4]=ブロックID(12bit) [3:0]=セルID(4bit)
                            // 開始位置を4KBブロック＋256Bセル単位で指定
    uint8_t  size[3];       // リソースサイズ（バイト・最大16MB）
    uint16_t res_type_id;   // [15:12]=リソース種別(4bit) [11:0]=リソースID(12bit)
    uint8_t  reserved;      // 予備
} ResourceEntry;  // 計8バイト・1ブロック4KBで512エントリ管理可能
```

**ノベルゲームで使用するリソース種別：**

| 種別値 | 内容 |
|--------|------|
| 0x0 | プログラム（バイトコードブロック） |
| 0x1 | 画像（背景・CG・スプライト） |
| 0x2 | 音声（BGM・VGMバイナリ） |
| 0x3 | 音声（効果音・ADPCM） |
| 0x5 | シナリオ（バイトコード列） |
| 0xF | FM音色定義拡張 |

**メタ情報領域レイアウト（TFT/OLED機種・6KB）：**

| オフセット | サイズ | 内容 |
|-----------|--------|------|
| 0x0000 | 4KB | アプリ内ブロックディレクトリ#0（512エントリ×8B） |
| 0x1000 | 1KB | FM音色定義（48音色・拡張可） |
| 0x1400 | 512B | シナリオ継続情報（SCN_PTR・フラグ・ENV_STATE等） |
| 0x1600 | 128B | アプリメタ情報（タイトル・作者・バージョン） |
| 0x1680 | 384B | 予備 |

**ブロック配置順序（PC側ビルドツールが保証）：**

```
ブロック0: プログラム（App Area分9KB＋メタ情報領域7KB）
ブロック1: FM音色拡張（固定1ブロック・不使用でも確保）
ブロック2〜N: ディレクトリ拡張（必要数分・Fate規模では複数必要）
ブロックN+1〜: データ（画像・音声・シナリオ等）
```

### 3.1c 共通プログラム（1層）API呼び出し

ノベルゲームインタプリタ（App Area）から共通プログラムの機能を使う場合、
固定アドレス0x00000804に配置されたLtcApiテーブル経由で呼び出す。

```c
#include "ltc_api.h"  // アプリ側にコピーして使用

// 表示（機種非依存）
LTC_API->disp_fill(DISP_BLACK);
LTC_API->disp_draw_string("こんにちは", 10, 10, DISP_WHITE, DISP_BLACK);

// リソースロード
LTC_API->load_resource(0x5001, fram_dest);  // シナリオID=1をFRAMに展開

// ブロック切り替え（パターンB）
LTC_API->iap_run(next_block_addr);  // 戻らない
```

### 3.2 1ブロック完結型IAP駆動シーケンス（ノベルゲーム専用方式）

数メガバイトに及ぶシナリオを、PC側コンパイラによって「1ブロック＝最大9KB以下」のバイナリへ機械的に細切れ分割する。ブロック切り替えはコンテキスト退避を行わない**ソフトリセット方式**を採用する。

これは汎用のIAPコンテキストスイッチ（SRAM退避・復帰あり）とは異なるノベルゲーム専用の最適化であり、SRAMの2KB退避コストが不要になる合理的な選択である。セーブデータおよびシナリオ継続に必要な情報はすべてFRAMに保持されるため、ソフトリセット後も状態は完全に維持される。

```
(1) インタプリタはApp Area内（10KB）のバイトコードを上から順にパース・実行

(2) シナリオの節目、または10KBブロックの末尾に達した時点で
    CMD_BLOCK_NEXT [SPI_FLASH_ADDR] 命令を検知

(3) インタプリタは共通プログラムの iap_run() を呼び出す前に
    以下をFRAMのセーブデータ領域に書き込む：
      - 次ブロックのSPI Flashアドレス（SCN_PTR）
      - 現在の環境状態（背景ID・BGM ID）
      - 長寿命フラグ群（FLAGS[0〜399]）

(4) iap_run() が内蔵SRAMで実行され
    SPI Flashの指定アドレスから次の10KBをApp Areaへ上書き

(5) ソフトウェアリセット発動

(6) 再起動後のインタプリタがFRAMのSCN_PTRを参照して
    続きのシナリオを何事もなかったかのように再生開始
```

ブロック切り替えはシナリオの節目（章末・シーン切り替え等）に配置するため、プレイヤーには自然な暗転・演出として見える。

## 4. セーブデータ構造設計（FRAM 16KB運用）

FRAMの固定領域（0x02000〜0x05FFF・16KB）を使用する。1スロット512バイトとして、最大28スロットを厳密に構築する。

### 4.1 スロット構成

|スロット種別      |スロット数 |サイズ   |用途            |
|------------|------|------|--------------|
|グローバルシステムセーブ|4スロット |2KB   |ゲーム全体の進捗・設定   |
|ユーザーセーブ     |24スロット|12KB  |プレイヤーの任意セーブ   |
|オートセーブ      |1スロット |0.5KB |ブロック切り替え時の自動保存|
|予備          |残り    |〜1.5KB|将来拡張用         |

### 4.2 1スロット（512バイト）の内部メモリマップ

|オフセット|サイズ |フィールド名   |内容                                         |
|-----|----|---------|-------------------------------------------|
|0x000|4B  |MAGIC    |破損チェック用ヘッダ `0x53, 0x41, 0x56, 0x45`（“SAVE”）|
|0x004|4B  |SCN_PTR  |ロード時にジャンプするSPI Flash内の絶対シナリオアドレス           |
|0x008|8B  |TIMESTAMP|セーブ日時・プレイ時間カウント（秒）                         |
|0x010|4B  |ENV_STATE|復帰用環境情報（現在の背景画像ID・BGM再生ID）                 |
|0x014|400B|FLAGS    |長寿命フラグ変数領域（8bit × 400個）                    |
|0x1A4|4B  |CRC32    |データ破損検知用チェックサム                             |
|0x1A8|92B |PADDING  |将来の拡張用予備領域                                 |

**合計: 512バイト**

### 4.3 PC側コンパイラによる変数の寿命最適化

内蔵SRAM（2KB）を圧迫しないため、PC側で元スクリプト（.ks等）の変数を静的解析し、寿命によって2種類に分離する。

**(1) 長寿命フラグ（セーブ対象）**

ルート確定・好感度・クリア状況など、物語全体にわたって影響する変数。0〜399番の固定アドレスへパッキングされ、FRAMのFLAGS領域に直接マップされる。IAPブロック切り替えをまたいで保持される。

**(2) 短寿命フラグ（セーブ非対象）**

「直前の選択肢の分岐」など、数行〜数ページで役割を終える使い捨て変数。FRAMには含めず、内蔵SRAM（2KB）のワークエリアで処理する。IAPブロック切り替え（ソフトリセット）時に自動的にゼロクリアされる。

## 5. アセット圧縮および変換ロジック

16MBのSPI Flashに全アセットを格納するため、PC側ビルドツールで以下の変換を行う。

### 5.1 画像アセット（背景・立ち絵・CG）

#### 背景・イベントCG

|処理    |内容                              |
|------|--------------------------------|
|解像度縮小 |元800×600を歪みなしで240×180（4:3を維持）へ縮小|
|減色    |インデックスカラー256色（8bit）へ減色          |
|圧縮    |uzlib（zlib互換）で圧縮                |
|想定サイズ |1枚あたり約5KB〜12KB                  |
|200枚換算|約2.4MB                          |

#### 立ち絵スプライト

透過全画面ではなく、キャラクターが描画されている最小矩形領域（バウンディングボックス）をPC側で自動クロップして格納する。実機側へは描画座標（X, Y）を指定して透過転送する形式を採り、SRAMバッファを節約する。

|処理   |内容                       |
|-----|-------------------------|
|クロップ |PC側でバウンディングボックスを自動検出・切り出し|
|描画方式 |座標（X, Y）指定で部分転送          |
|想定サイズ|全スプライト計 約3.0MB           |

### 5.2 音楽・音声アセット

#### BGM

PCMオーディオ再生はリソース的に不可能なため、MIDI/MMLデータを経由してYMF825の直接レジスタ制御命令列（VGMバイナリ形式）へ全自動シリアライズする。

|項目      |内容                      |
|--------|------------------------|
|形式      |VGMバイナリ（YMF825レジスタ制御命令列）|
|1曲あたりサイズ|10KB〜30KB               |
|40曲換算   |約800KB                  |

#### 効果音（SE）

基本はYMF825のFM音源ノイズ・エフェクトチャンネルで模倣する。生音が必要なもののみADPCMで保持する。

|種別           |方式                      |サイズ             |
|-------------|------------------------|----------------|
|通常SE         |YMF825 FMチャンネル模倣        |演奏データのみ（数十バイト/音）|
|生音SE（風切り・打撃等）|8kHz / 4bit ADPCM（PWM再生）|1〜2秒で数KB        |
|合計（全SE）      |—                       |約200KB          |

## 6. 実機インタプリタ（3層）コマンド設計

内蔵Flash 10KBのApp Areaに収めるため、インタプリタはプレーンテキストの文字列解析を一切行わず、1〜2バイト単位の固定長バイトコードのみを高速評価するステートマシンとして実装する。

### 6.1 バイトコード命令セット

|オペコード|命令名            |引数                                       |動作                          |
|-----|---------------|-----------------------------------------|----------------------------|
|0x01 |CMD_TEXT_DISP  |[文字数(1B)] [恵梨沙コード配列(2B×N)]               |テキスト描画                      |
|0x03 |CMD_CLICK_WAIT |なし                                       |キー入力待ち。最下行に「▼」明滅            |
|0x05 |CMD_BG_CHANGE  |[画像ID(1B)]                               |SPI Flashから背景を読み込み上部180pxに描画|
|0x07 |CMD_SPRITE_SHOW|[キャラID(1B)] [X座標(1B)] [Y座標(1B)]          |立ち絵表示（バウンディングボックス転送）        |
|0x09 |CMD_BGM_PLAY   |[楽曲ID(1B)]                               |YMF825への演奏ストリーム開始           |
|0x0B |CMD_FLAG_SET   |[変数Addr(2B)] [演算子(1B)] [値(1B)]           |フラグ操作（長寿命: FRAM / 短寿命: SRAM）|
|0x0D |CMD_FLAG_JUMP  |[評価変数Addr(2B)] [条件(1B)] [ブロック内相対Addr(2B)]|条件分岐                        |
|0x0F |CMD_BLOCK_NEXT |[次ブロックのSPI Flash Addr(4B)]               |FRAMへ状態保存後、iap_run()発動      |

### 6.2 インタプリタのサイズ見積もり

App Area 10KB内に以下を収める。

|モジュール                         |目標サイズ     |
|------------------------------|----------|
|ステートマシン本体（バイトコード評価ループ）        |〜4KB      |
|テキスト描画（恵梨沙フォント参照・FRAM読み出し）    |〜2KB      |
|画像転送（SPI Flash読み出し・TFT部分Flush）|〜1.5KB    |
|セーブ・ロード処理（FRAM読み書き）           |〜1KB      |
|オーディオ再生機能（YMF825・ADPCM）       |〜1KB      |
|**合計**                        |**〜9.5KB**|

## 7. PC側ビルドツール構成

### 7.1 処理パイプライン

```
元スクリプト（.ks / .rpy / .nscript 等）
    ↓
[1] 静的解析・変数分類
    長寿命フラグ → FLAGSアドレス割り当て（0〜399）
    短寿命フラグ → SRAMワークエリアアドレス割り当て
    ↓
[2] バイトコードコンパイル（トランスパイル）
    元スクリプトの命令 → バイトコード列に変換
    10KB境界でCMD_BLOCK_NEXT挿入
    ↓
[3] アセット変換
    背景・CG: 縮小 → 減色（256色）→ uzlib圧縮
    立ち絵:   バウンディングボックスクロップ
    BGM:      MIDI/MML → VGMバイナリシリアライズ
    SE:       WAV → 8kHz/4bit ADPCM変換
    ↓
[4] SPI Flashイメージ生成
    カタログ + 恵梨沙フォント + バイトコードブロック群 + アセット群
    → 16MB SPI Flashイメージ（.bin）
    ↓
[5] USB HID経由でSPI Flashへ書き込み
    （ドライバ不要・Windows/Linux/macOS/Android対応）
```

## 8. 技術的結論と現実性の証明

Fate/stay nightを素材とした場合の想定アセット総容量推定：

|アセット種別          |推定サイズ     |
|----------------|----------|
|シナリオバイトコード列     |約500KB〜1MB|
|恵梨沙フォント（固定）     |55KB      |
|背景・イベントCG（圧縮済）  |約3.5MB    |
|立ち絵スプライト（クロップ済） |約3.0MB    |
|BGM演奏レジスタ配列（VGM）|約800KB    |
|効果音（ADPCM / FM） |約200KB    |
|**実効総合計**       |**約9.0MB**|

これはCAR-01が標準搭載する**16MB SPI Flashの約56%**に過ぎない。残り44%（約7MB）には以下を収容できる。

- 複数ルートの追加バイトコード
- 追加CGおよびスプライトバリエーション
- 追加BGMトラック
- 他タイトルの収録（複数タイトル同梱も視野に入る容量）

**結論**: 「16KB Flash / 2KB SRAMの極小MCU上で、世界最高峰の巨大ノベルゲームを完全ネイティブ駆動させる」というアプローチは、容量的・ロジック的に完全な現実解である。
