## 3. App Areaの制約

**App Area：0x1E00〜0x3FFF（8KB）・全機種統一**

アプリ本体コードはこの8KB以内に収める。IAPにより任意のタイミングで内容が切り替わる。

```
App Area（8KB）
├── アプリ本体コード（ゲームロジック・インタプリタ等）
├── オーディオ再生機能（〜1KB目標・YMF825制御）
└── （IAPコンテキストスイッチを使う場合）アセンブリスタブ
```

**メタ情報領域（6KB・App Areaの直後）：**
SPI Flash上のアプリブロックに付随するメタデータ領域。コンバータが自動生成する。

| オフセット | サイズ | 内容 |
|---|---|---|
| 0x0000 | 4KB | アプリ内ブロックディレクトリ#0（512エントリ×8B・プログラム専用） |
| 0x1000 | 32B | リソース種別開始アドレス表（type_start[16]） |
| 0x1020 | 約1KB | FM音色定義（48音色・種別0xF） |
| 0x1400 | 512B | シナリオ継続情報（SCN_PTR・フラグ群・パターンBソフトリセット用） |
| 0x1600 | 128B | アプリメタ情報（タイトル・作者・バージョン・ビルド日時） |
| 0x1680 | 64B | IAP A-1引数エリア（INTERNAL） |
| 0x16C0 | 64B | IAP A-1戻り値エリア（INTERNAL） |
| 0x1700 | 64B | IAP A-2引数エリア（EXTERNAL） |
| 0x1740 | 64B | IAP A-2戻り値エリア（EXTERNAL） |
| 0x1780 | 128B | 予備 |

## 4. RcApiの使い方

共通プログラム（1層）の機能はすべてRcApiテーブル（固定アドレス **0x08A0**）経由で呼び出す。

```c
// rc_api.h（アプリが#includeする）
#include "rc_api.h"

// 使用例
void my_app_main(void) {
    // ディスプレイ初期化
    RcApi.disp_init();

    // 全面塗りつぶし
    RcApi.disp_fill(DISP_BLACK);

    // 文字列描画
    RcApi.disp_draw_string(10, 20, "Hello!", DISP_WHITE);

    // ボタン待機
    RcApi.keyscan_wait(JOY_CENTER);

    // アプリ切り替え（戻らない）
    RcApi.iap_run(NEXT_BLOCK_ADDR);
}
```

**ディスプレイ独立性：** disp_*関数はTFT・e-ink・OLED全機種で同一APIを呼び出す。機種ごとの差異は共通プログラム内部で吸収される。

## 5. リソース管理

### リソースエントリ構造体（8バイト固定）

```c
typedef struct {
    uint16_t res_type_id;   // [15:12]=種別(4bit) [11:0]=ID(12bit)
    uint16_t block_cell;    // [15:4]=ブロックID(12bit) [3:0]=セルID(4bit)
    uint8_t  size[3];       // サイズ（バイト・最大16MB）
    uint8_t  reserved;
} ResourceEntry;
```

### リソース種別

| 種別値 | 内容 |
|---|---|
| 0x0 | プログラム（IAPで別モジュールをロード） |
| 0x1 | 画像 |
| 0x2 | 音声（BGM） |
| 0x3 | 音声（効果音） |
| 0x4 | 音声（ボイス） |
| 0x5 | シナリオ |
| 0xE | ディレクトリ拡張ブロック |
| 0xF | FM音色定義拡張ブロック |

### リソースのロード

```c
// SPI Flash上のリソースをFRAM作業領域にロード
// res_type_id: [15:12]=種別 [11:0]=ID
// fram_dest_addr: FRAM作業領域上の展開先アドレス
RcApi.load_resource(0x1001, FRAM_WORK_BASE + 0x8000);  // 画像ID=1をロード
```

**重ね合わせはアプリの責務。** 共通プログラムはロード・転送まで担い合成には関与しない。

### アプリ規模の目安

| 規模 | 対象グレード | ディレクトリブロック数 |
|---|---|---|
| シンプル（小規模ゲーム） | スタンダード | 1ブロック（〜512エントリ） |
| 中規模ゲーム | スタンダード〜プレミアム | 1〜2ブロック |
| 大規模（Fate規模・声なし） | プレミアム | 2〜3ブロック（1,000〜1,500リソース） |
| 超大規模（Fate規模・声あり） | プレミアム | 数十ブロック（数千〜1万リソース） |

## 6. IAPパターンの使い分け

### パターンB：ソフトリセット（推奨・シンプル）

戻らない。状態をFRAMに明示保存して次ブロックで参照する。ノベルゲーム・シナリオ連続再生に適する。

```c
// 状態をFRAMのシナリオ継続情報に保存
fram_write(SCN_PTR_ADDR, &next_addr, sizeof(next_addr));

// 次ブロックに切り替え（戻らない）
RcApi.iap_run(next_block_flash_addr);
```

### パターンA：コンテキストスイッチ（上級・複雑）

戻る。SRAMイメージをFRAMに退避して復元する。大規模アプリのモジュール分割に適する。

```c
// 別モジュールを呼び出す（戻る）
RcApi.iap_call(module_flash_addr, MODULE_ID, IAP_CALL_INTERNAL);

// モジュール側でiap_return()を呼ぶと元のブロックに戻る
```

**所要時間：** 約117〜290ms。FEPのような頻繁な呼び出しには不向き。場面転換・シーン切り替えなど「イベント」として呼ぶ用途が適切。

## 7. オーディオ実装方針

**オーディオ機能は3層（アプリブロック組み込み）。** コンテキストスイッチ中は音声が途絶えるため1層・2層には含めない。

```
App Area（8KB）内:
├── アプリ本体コード
└── オーディオ再生機能（〜1KB目標）
    ├── YMF825 SPI送信ラッパー（CS_YMF=PD2）
    ├── FM音源再生コマンド
    └── ADPCM/PWM再生（AUDIO_IN経由・UIAPduino PC3）
```

**YMF825初期化：** アプリ起動時に初期化ブロック（IAPで呼び出し）でYMF825の初期化・音色設定を完了させる。アプリ本体からはレジスタ書き込みのみ行う（再初期化不要）。

**音量制御：** YMF825マスターボリュームレジスタ（0x19）をUIAPduino経由でATtiny1604（ジョイスティック操作）からソフトウェア制御する。詳細は `features_system_menu_20260615.md` §4参照。

## 8. 日本語表示

恵梨沙フォント（6877文字・各8B）はブートローダ起動時にSPI Flash（0x008000）からFRAM作業領域へ自動展開される（約18ms）。アプリ開発者はフォントロード処理を書く必要はない。

```c
// 恵梨沙フォントindexからFRAMアドレスを計算
uint32_t char_addr = FONT_FRAM_BASE + (elysia_index * 8);

// 文字データをFRAMから読み出してディスプレイに描画
RcApi.disp_draw_string_elysia(x, y, elysia_index_array, len, color);
```

**タイトル文字列：** CatalogEntryのtitleフィールドは恵梨沙フォントindexの配列（uint16_t×8文字）。UTF-8やShift-JISではなくインデックスで格納する。PC側ビルドツールが変換を担当する（PENDING #1）。

## 9. ノベルゲーム開発

バイトコード方式のノベルゲームインタプリタをApp Areaに組み込む構成。

**参考実装（Fate/stay night規模・声なし）：**

```
App Area（8KB）:
├── ノベルゲームインタプリタ本体
└── オーディオ再生機能（〜1KB）

SPI Flash上:
├── シナリオデータ（バイトコード列）
├── 背景画像・CG
├── BGM（YMF825 VGMバイナリ）
└── 効果音（ADPCM）
```

**ブロック切り替え（パターンB）：**

```
(1) インタプリタがCMD_BLOCK_NEXT [SPI_FLASH_ADDR]命令を検知
(2) FRAMのシナリオ継続情報に以下を書き込む
    ・次ブロックのSPI Flashアドレス（SCN_PTR）
    ・継続に必要な環境状態（背景ID・BGM ID等）
    ・長寿命フラグ群
(3) iap_run()で次のバイトコードブロックをApp Areaに上書き
(4) ソフトウェアリセット
(5) 再起動後のインタプリタがFRAMのSCN_PTRを参照して続きから再開
```

**シナリオフォーマット：** バイトコード方式で詳細は未決定（PENDING #2）。GB Studio・NScripter・吉里吉里KAGを参考として設計する。
