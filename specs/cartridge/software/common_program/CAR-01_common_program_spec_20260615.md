## 2. 3層アーキテクチャ概要

CAR-01（UIAPduino版）のソフトウェアは3層で構成される。

```
┌─────────────────────────────────────────┐
│  3層: アプリ（App Area 8KB）             │
│  開発者が実装。ゲーム・ツール・ノベル等    │
│  RcApi（1層）・2層ライブラリを呼び出す    │
├─────────────────────────────────────────┤
│  2層: 共通モジュール（SPI Flash上）       │
│  IAPコンテキストスイッチで呼び出す         │
│  セーブデータAPI・描画拡張ライブラリ等     │
│  ※コンテキストスイッチ中は音声が途絶える   │
├─────────────────────────────────────────┤
│  1層: 共通プログラム（内蔵Flash固定）     │
│  起動シーケンス・RcApi・IAP・SPIマネージャ │
│  アドレス固定（0x0800〜0x1BFF）          │
└─────────────────────────────────────────┘
```

| 層 | 提供形態 | 呼び出し方 | 用途 |
|---|---|---|---|
| **1層** | 内蔵Flash固定（0x0800〜0x1BFF） | RcApiテーブル経由で直接呼び出し | 起動・IAP・キースキャン・表示最小・リソースロード |
| **2層** | SPI Flash上（IAPで呼び出し） | iap_call()でロード後に呼び出し | 機能リッチな描画・セーブデータAPI・Arduboy互換レイヤー |
| **3層** | App Area（IAPで書き換え） | 直接実装 | ゲームロジック・オーディオ・日本語入力エンジン |

**オーディオの位置づけ：** コンテキストスイッチ中は音声が途絶えるため、オーディオ機能は3層（アプリブロック組み込み）とする。〜1KBを目標サイズとする。YMF825の初期化・音色設定はアプリ起動時の初期化ブロックで実施する。

## 3. 内蔵Flashレイアウト

CH32V003の内蔵Flash 16KBは全機種（TFT・OLED・e-ink）で統一レイアウトを採用する。

```
0x0000 ┌─────────────────────────┐
       │ UIAPduinoブートローダ   │ 2KB枠（0x0000〜0x07FF）
       │                         │ 実測1,892B・計算基準2,048B
0x0800 ├─────────────────────────┤ ← リンカORIGIN・LENGTH=5632
       │ ラテカ共通プログラム    │ 5.5KB枠（0x0800〜0x1BFF）
       │ UIAPduino2KB込み合計    │ 7.5KB
0x1C00 ├─────────────────────────┤
       │ 共通モジュール置き場    │ 512B（0x1C00〜0x1DFF）
       │ （開発者自由領域）      │ IAPによるApp Area書き換えで上書きされない
0x1E00 ├─────────────────────────┤
       │ App Area（3層）         │ 8KB（0x1E00〜0x3FFF）
0x3FFF └─────────────────────────┘
```

| 領域 | アドレス | サイズ | 書き換え手段 |
|---|---|---|---|
| UIAPduinoブートローダ | 0x0000〜0x07FF | 2KB枠 | セルフアップデートのみ |
| 共通プログラム（1層） | 0x0800〜0x1BFF | **5.5KB枠** | USB HID経由 |
| 共通モジュール置き場 | 0x1C00〜0x1DFF | **512B** | write_program.py等で書き込む |
| App Area（3層） | 0x1E00〜0x3FFF | **8KB** | IAPで書き換え |

**実測値（確定）：**
- 共通プログラム単体：**5,140B**
- UIAPduino込み合計：**7,188B**
- 5.5KB枠（5,632B）に対する余裕：**492B** ✅
- RcApiテーブル配置：**0x08A0**（割り込みベクタテーブル160B直後）

**CH32V003の3層ブートローダ構造：**

| 層 | 場所 | 書き換え手段 | 備考 |
|---|---|---|---|
| System Flash | 別物理領域 | 不可能 | WCH工場固定・UARTブートローダ |
| UIAPduinoブートローダ | 0x0000〜0x07FF | セルフアップデート機能のみ | rv003usbベース・USB HID実現 |
| 共通プログラム（1層） | 0x0800〜0x1BFF | USB HID経由 | 通常の開発サイクル |
| App Area（3層） | 0x1E00〜0x3FFF | IAP（共通プログラムが担当） | SPI Flashから書き換え |

## 4. 起動シーケンス

```
【電源投入】
    ↓
UIAPduinoブートローダ起動（USB書き込み待機・タイムアウト後0x0800へジャンプ）
    ↓
boot_main()（共通プログラム起点）
    ↓
キースキャン処理を呼び出し
    ↓
    ├─ 上下キー同時押し → ブートローダセルフアップデートモードへ（§5.6参照）
    │
    ├─ 決定ボタン未押下
    │       ↓
    │   App Areaにアプリが存在するか確認
    │       ├─ 存在する → 恵梨沙フォントロード → App Areaに制御を移す
    │       └─ 存在しない → 決定ボタン押下時と同じフローへ
    │
    └─ 決定ボタン押下
            ↓
        200ms待機（キーボード側GPIO2セットアップの猶予）
        GPIO2確認 → HIGH: 外部通信モード（SPI Flash管理処理を起動）
                  → LOW:  アプリロード処理へ
            ↓
        FRAMの管理領域からアプリ一覧を読み込み
        TFT/e-inkにアプリタイトルリストを表示
        キースキャン: 上下で選択・決定で確定
            ↓
        RAM展開型IAPを呼び出し（選択した先頭ブロック番号を引数）
        SPI Flashからプログラムブロック（App Area 8KB）を焼く
        メタ情報領域（6KB）をFRAM作業領域に転送
            ↓
        ソフトウェアリセット

【再起動（アプリ起動フロー）】
    ↓
ブートローダ → boot_main()
    ↓
恵梨沙フォントをSPI Flash（0x008000）→ FRAM作業領域へ転送（約55KB・約18ms）
← この時点から日本語表示が可能になる
    ↓
決定ボタン未押下 → App Areaに制御を移す → アプリ起動
```

## 5. 1層：共通プログラム仕様

### 5.1 モジュール構成

| モジュール | 実測サイズ | 内容 |
|---|---|---|
| UIAPduinoブートローダ | 2KB | 起動プロセス呼び出し分岐追加版 |
| boot.c | — | アプリ選択UI・起動フロー分岐 |
| app_loader.c | 〜1KB | FRAMキャッシュからアプリ一覧取得・表示・選択 |
| spi_manager.c | 〜0.5KB | ハードウェアSPI直接制御・CS4系統切り替え |
| tft_oled.c または eink.c | 〜1KB / 〜3KB | 最小ドライバ・初期化シーケンス（機種別） |
| iap.c | 〜0.5KB | RAM展開型IAP・iap_run/call/return |
| keyscan.c | 〜1KB | I2C経由ジョイスティック入力読み取り |
| font_elysia.c | 〜0.2KB | 英数ビットマップフォント（余剰領域） |
| rc_api.c | — | RcApiテーブル（0x08A0固定配置） |
| **合計** | **5,140B（確定）** | |

### 5.2 RcApiテーブル（固定アドレスAPI）

RcApiは固定アドレス **0x08A0** に配置される12関数のAPIテーブル。アプリ（3層）はこのテーブルを通じて共通プログラムの機能を呼び出す。アドレスが固定のため、共通プログラムを更新してもアプリのバイナリを再コンパイルせずに利用できる。

**ディスプレイ非依存（機種共通）：**

```c
iap_run(uint32_t flash_addr)
    // アプリ切り替え（パターンB・戻らない）

iap_call(uint32_t flash_addr, uint8_t caller_id, uint8_t call_type)
    // モジュール呼び出し（パターンA・戻る）
    // call_type: IAP_CALL_INTERNAL=0 / IAP_CALL_EXTERNAL=1

iap_return(void)
    // 呼び出し元に戻る

load_resource(uint16_t res_type_id, uint32_t fram_dest_addr)
    // SPI Flash→FRAM作業領域へリソースロード

keyscan_get(void)
    // ボタン状態取得（ビットマスク）

keyscan_wait(uint8_t mask)
    // 指定ボタン押下待機

keyscan_get_button(uint8_t button)
    // 特定ボタン確認
```

**ディスプレイ依存（機種ごとに実装が異なる）：**

```c
disp_init(void)          // 初期化
disp_fill(DispColor c)   // 全面塗りつぶし
disp_draw_string(...)    // 文字列描画（恵梨沙フォント・位置指定込み）
disp_set_window(...)     // 描画ウィンドウ指定
disp_write_pixel(...)    // 1ピクセル書き込み（→2層移管を検討）
```

**色定数（uint16_t統一）：**

```c
typedef uint16_t DispColor;
#define DISP_BLACK  0x0000U  // 全機種: 黒
#define DISP_WHITE  0xFFFFU  // 全機種: 白
#define DISP_RED    0xF800U  // TFT: 赤 / e-ink4色: 赤 / 白黒: 黒寄り
#define DISP_GREEN  0x07E0U  // TFT: 緑 / 白黒: 白寄り
#define DISP_BLUE   0x001FU  // TFT: 青 / 白黒: 黒寄り
```

**API公開方針：**

| カテゴリ | 公開先 | 理由 |
|---|---|---|
| RcApiテーブル（12関数） | 全アプリ向け | 機種非依存の標準API・固定アドレス0x08A0 |
| FRAM低レベルI/O（fram_write等） | 共通プログラム内部専用 | アプリによる誤書き込み・破壊を防ぐ |
| SPI Flash低レベルI/O（flash_read等） | 共通プログラム内部専用 | カタログ・フォント領域の保護 |
| セーブデータAPI | 2層ライブラリとして提供 | FRAM保護領域（0x02000〜0x05FFF）に限定 |

### 5.3 IAPパターン

CAR-01のIAPには3パターンがある。

| パターン | 名称 | 戻り | 用途 |
|---|---|---|---|
| B | ソフトリセット（1ブロック完結型） | 戻らない | ノベルゲーム・シナリオ連続再生 |
| A-1 | 同一アプリ内モジュール呼び出し | 戻る | 大規模アプリのモジュール分割 |
| A-2 | 別アプリ呼び出し | 戻る | アプリ間サービス呼び出し |

**所要時間（推定・CH32V003 @48MHz）：**

| フェーズ | 時間 |
|---|---|
| 内蔵SRAM 2KB → FRAM退避 | 0.7ms |
| SPI Flashからモジュール読み出し（8KB） | 5.5ms |
| 内蔵Flash消去（4セクタ） | **80ms**（ボトルネック） |
| 内蔵Flash書き込み（8KB） | **26ms** |
| ソフトリセット〜再起動 | 5ms |
| FRAMからSRAMイメージ復元 | 0.7ms |
| **合計（標準）** | **≈117ms** |
| **合計（最悪）** | **≈290ms** |

### 5.4 IAPコンテキストスイッチ（パターンA）

**コンテキストスタック構造：**

```c
typedef struct {
    uint8_t  restore_flag;       // 1=有効
    uint8_t  module_id;          // 呼び出し元モジュールID
    uint8_t  call_type;          // IAP_CALL_INTERNAL=0 / IAP_CALL_EXTERNAL=1
    uint8_t  pad;
    uint32_t sp;                 // 呼び出し元スタックポインタ
    uint32_t ra;                 // 呼び出し元戻りアドレス
    uint32_t caller_flash_addr;  // 呼び出し元ブロックのSPI Flashアドレス
    uint8_t  sram_image[2048];   // 内蔵SRAM 2KBスナップショット
} ContextEntry;  // 計2,062B
```

**スタック配置（FRAM作業領域先頭）：**

| 項目 | サイズ | 内容 |
|---|---|---|
| スタック深さカウンタ | 1B | 現在のネスト段数 |
| エラーコード・予備 | 15B | デバッグ情報等 |
| コンテキスト[0]〜[6] | 2,062B × 7 = 14,434B | 各段のSRAMイメージ+復帰情報 |
| **合計** | **約14.5KB** | |

**最大ネスト段数：7段**（8段目は16,480Bとなり16KBブロックを超過するため不可）

**IAP引数・戻り値エリア（FRAM固定領域）：**

| 領域 | FRAMアドレス | サイズ | 用途 |
|---|---|---|---|
| IAP A-1引数（INTERNAL） | 0x16680 | 64B | 同一アプリ内モジュール呼び出し引数 |
| IAP A-1戻り値（INTERNAL） | 0x166C0 | 64B | 同上・戻り値 |
| IAP A-2引数（EXTERNAL） | 0x16700 | 64B | 別アプリ呼び出し引数 |
| IAP A-2戻り値（EXTERNAL） | 0x16740 | 64B | 同上・戻り値 |

### 5.5 SPI Flash管理処理

FRAMの固定領域（0x08000〜0x0FFFF・32KB）に格納する独立したプログラム。boot_main()がGPIO2=HIGHを検出した場合にIAP経由で内蔵Flashに書き込まれ実行される（外部通信モード）。

**役割：**

| 役割 | 内容 |
|---|---|
| PCとのデータ交換 | USB HID経由でSPI Flashの読み書き消去を行う |
| カタログ管理 | SPI Flash上のアプリカタログを更新する |
| FRAMキャッシュ更新 | カタログ変更後にFRAMの管理領域キャッシュを更新する |

**HIDレポートフォーマット（確定）：**
64B固定（USB Full Speed物理制約）：`cmd(1B) / seq(1B) / addr(4B) / length(2B) / data(56B)`

### 5.6 ブートローダセルフアップデート

boot_main()起動時に**上下キー同時押し**で呼び出す。WCH-LinkEを不要とするための仕組み。

**SPI Flash格納場所：** システム予約ブロック内1ブロック先頭4KB

```
0x000 ┌──────────────────────┐
      │ アップデートプログラム │ 〜512B（SRAM展開型）
0x200 ├──────────────────────┤
      │ 新ブートローダバイナリ │ 2KB（0x0000〜0x07FFに書く内容）
0xA00 ├──────────────────────┤
      │ チェックサム（CRC32） │ 4B
0xFFF └──────────────────────┘
```

**動作フロー：**

```
上下キー同時押し検出
    ↓ SPI Flashからアップデートプログラム（〜512B）をApp Area（0x1E00〜）にIAPで書き込む
    ↓ ソフトウェアリセット → アップデートプログラムが起動
    ↓ CRC32検証 → 新ブートローダをSRAMに読み込む
    ↓ __disable_irq()
    ↓ Flash書き換えルーチンをSRAM上に展開・実行
    ↓ 0x0000〜0x07FFを消去 → 新ブートローダで上書き
    ↓ App Area（0x1E00〜0x3FFF）を消去
    ↓ ソフトウェアリセット → 新ブートローダ起動
```

## 7. FRAMレイアウト

1チップ構成。グレードで容量のみ異なり、アドレス構造は共通。

```
FRAM（256KB or 512KB）
0x00000 ┌──────────────────────┐
        │ 管理領域             │ 8KB
        │                      │ SPI Flashカタログヘッダキャッシュ
        │                      │ キャッシュ: 26B × 58エントリ = 1,508B
0x02000 ├──────────────────────┤
        │ セーブデータ         │ 16KB（0x02000〜0x05FFF）
0x06000 ├──────────────────────┤
        │ 共通プログラム領域   │ 8KB（自己復元用バックアップ）
0x08000 ├──────────────────────┤
        │ SPI Flash管理処理    │ 32KB（固定）
0x10000 ├──────────────────────┤
        │ SRAM相当作業領域     │ プレミアム版: 448KB
        │                      │ スタンダード版: 192KB
```

**SRAM相当作業領域の内訳（確定版・2026-06-24）：**

```
0x10000 ┌──────────────────────┐
        │ コンテキストスタック  │ 14,464B（ヘッダ16B + ContextEntry 2,064B × 7）
0x13880 ├──────────────────────┤
        │ 恵梨沙フォント展開   │ 64KB確保（実データ約55KB・6877文字×8B）
0x23880 ├──────────────────────┤
        │ アプリ自由作業領域   │ プレミアム: 約364KB
        │ （FRAM_META_BASE）   │ スタンダード: 約108KB
```

**FRAMアドレス正式値（2026-06-24確定）：**

| 領域 | 開始アドレス | サイズ | 根拠 |
|---|---|---|---|
| コンテキストスタック（CTX_STACK_BASE） | 0x10000 | 14,464B | ヘッダ16B + ContextEntry(2,064B)×7 |
| 恵梨沙フォント展開（FONT_FRAM_ADDR） | 0x13880 | 64KB（0x10000） | 実データ55KB + 拡張余裕 |
| アプリ自由作業領域（FRAM_META_BASE） | 0x23880 | 残り全て | FONT_FRAM_ADDR + 64KB |

**UIAPduino交換方針：** UIAPduino本体（約290円）はFlash書き換え上限（約10,000回）に達したら交換する。FRAMとSPI Flashにすべてのデータが保全されているため、挿し替えだけで自動復元される（共通プログラムをFRAM 0x06000から自己IAPで復元）。

## 8. SPI Flashレイアウト

```
SPI Flash 16MB（W25Q128JVSIM）
0x000000 ┌──────────────────────┐
         │ カタログテーブル      │ 32KB固定（58エントリ × 538B）
         │                      │ 起動時にFRAM管理領域へキャッシュ

         │ 恵梨沙フォント        │ 64KB固定（6877文字×8B=55,016B）
         │ （固定アドレス）      │ ブートローダ起動時にFRAM作業領域へ展開（約18ms）

         │ システム予約領域      │ 512KB（ブートローダアップデート用等）
0x098000 ├──────────────────────┤
         │ アプリ領域            │ 約15.4MB・4KBブロック×3968本
         │ 4KBブロック単位管理   │ Wnn辞書（6.01MB）含む
0xFFFFFF └──────────────────────┘
```

**カタログエントリ構造体（確定版）：**

```c
typedef struct {
    uint8_t  valid;              // 0xFF=空き / 0x01=有効 / 0x00=論理削除
    uint8_t  disp_type;          // 0x01=e-ink / 0x02=TFT・OLED
    uint16_t title[8];           // 恵梨沙フォントインデックス列（最大8文字）
    uint32_t flash_addr;         // SPI Flash上の先頭アドレス（4KBアライン）
    uint32_t total_size;         // 全体サイズ（バイト）
    uint8_t  block_bitmap[512];  // 4096ブロック×1bit・重複検知用
} CatalogEntry;  // 計538B・58エントリ=31,204B≈30.5KB（32KB枠内）
```

### リンカスクリプト

```
MEMORY {
    FLASH (rx) : ORIGIN = 0x00000800, LENGTH = 5632  /* 5.5KB */
    RAM (xrw)  : ORIGIN = 0x20000000, LENGTH = 2048  /* 2KB */
}
```

### 実装制約

| 制約 | 内容 |
|---|---|
| Flash使用量 | UIAPduino込み7.5KB以内（共通プログラム5.5KB枠） |
| RAM使用量 | 内蔵SRAM 2KB以内（.data + .bss + スタック） |
| HAL不使用 | レジスタ直叩きのみ（ch32funのHALは不使用） |
| .ram_func | Flash書き換えルーチンはSRAMで実行 |

### 推奨実装順序

| Step | モジュール | 確認方法 |
|---|---|---|
| 1 | spi_manager.c | FRAM・SPI Flashへの単純Read/Writeが通るか確認 |
| 2 | keyscan.c | I2CでジョイスティックのATtiny1604から入力が読めるか確認 |
| 3 | tft_oled.c | 初期化→文字列1行表示が動くか確認（TFT ST7789から着手） |
| 3b | eink.c | e-ink調達後に実装 |
| 4 | app_loader.c | FRAMキャッシュからアプリ一覧が読めてTFTに表示できるか確認 |
| 5 | iap.c | SPI FlashのブロックをApp Areaに焼いて再起動できるか確認 |
| 6 | boot.c | ボタン判定→app_loader or App Area起動の分岐が正しく動くか確認 |
| 7 | SPI Flash管理処理 | USB HID経由でPCからSPI Flashへ書き込めるか確認 |
| 8 | font_load_elysia() | 恵梨沙フォントがFRAM作業領域に展開されTFTで日本語が表示できるか確認 |
| 9 | load_resource() | リソースIDを指定してSPI FlashからFRAM作業領域にロードできるか確認 |
