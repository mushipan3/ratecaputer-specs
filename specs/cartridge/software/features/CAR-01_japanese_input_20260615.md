## 1. 概要と設計方針

本書はラテカピュータ・カートリッジ（CAR-01）において、カートリッジ単体で日本語かな漢字変換入力を完結させるサブシステムの仕様を定める。

本システムはキーボードを前提とせず、カートリッジ単体で完結する自立稼働マシンとして設計する。日本語入力はカートリッジに内蔵されたジョイスティック（ATtiny1604 I2C化）のみで操作を完結する。

### 1.1 役務分担アーキテクチャ（ファットマスター・シンスレーブ・シンエンジン）

| 担当 | デバイス | 処理内容 |
|---|---|---|
| フロントエンド全般 | ATtiny1604（I2Cスレーブ） | ジョイスティック入力・チャタリング除去・ひらがな文字列生成・OLED候補表示・ユーザ確定操作 |
| 変換エンジンコア | UIAPduino CH32V003（App Area・IAPブロック） | 確定ひらがな列を受け取り辞書引き・Viterbi計算・候補列をFRAM掲示板へ書き込む。UIは一切持たない |
| 辞書前処理 | PC（Python・一度きり） | IPADIC→恵梨沙フォントindexへの変換・固定インデックス化・品詞フラグ化・SPI Flash用バイナリ生成 |
| 呼び出し元アプリ | UIAPduino CH32V003（App Area） | iap_call()でWnnブロックを起動し、IAP引数・戻り値エリアから確定文字列を受け取って画面表示する |

## 2. 処理フロー

```
呼び出し元アプリ（App Area・8KB）
    ↓ iap_call(EXTERNAL)  ← Flashロード（約110ms・ユーザ入力中に完了）

Wnnブロック（最大4KB枠）
    ↓ IAP_ARG_EXTERNALからひらがな文字列受取
    ↓ SPI Flash辞書引き（乗算1発でアドレス計算）
    ↓ Viterbi（単純変換はスキップ可・INTERNAL iap_callでネスト）
    ↓ IAP_RET_EXTERNALへ候補フォントindex列を書込
    ↓ iap_return()  ← FRAMからSRAMイメージ復元のみ（約0.7ms・Flashロードなし）

呼び出し元アプリ ← IAP掲示板から確定文字列を読んで画面表示

ATtiny1604（I2C並行動作）
    Wnn起動中: OLEDに「変換中」表示・次の入力待機
    iap_return()後: 候補列を受信してOLEDにスクロール表示
    ユーザがプッシュ確定 → 選択indexをUIAPduinoへI2C返送
```

### 2.1 体感時間の設計

| フェーズ | 時間 | ユーザ体感 |
|---|---|---|
| Flashロード（iap_call） | 約110ms | ユーザがひらがな入力中に完了・体感ゼロ |
| 辞書検索＋Viterbi | 約0.1ms | 無視できる |
| I2C候補データ転送（〜376B） | 約10ms | 体感ゼロ |
| OLED候補表示（ATtiny1604） | 瞬時 | 体感ゼロ |
| iap_return（FRAMからSRAM復元） | 約0.7ms | 体感ゼロ |
| ユーザ体感の支配項 | 入力速度そのもの | 変換ボタン→候補表示は瞬時 |

## 3. Wnn変換アルゴリズム

### 3.1 処理の流れ

1. **ラティス構築：** 入力ひらがな列に対し辞書を参照して形態素ネットワークを展開
2. **コスト割当：** 単語コスト（出現頻度）と接続コスト（品詞間連接）の総和を各経路に割当
3. **Viterbi探索：** 最小コスト経路を前方計算・後方追跡で一括決定

### 3.2 カートリッジ環境での課題と対処

| 課題 | 対処 |
|---|---|
| 組み合わせ爆発（長文） | ビーム幅制限（ビーム幅=1で単純変換・複雑時のみViterbiネスト） |
| 辞書が巨大（数MB） | SPI Flash 16MBに格納・固定インデックスで乗算1発アクセス |
| RAM 2KBでは展開不可 | FRAM 512KBをSRAM代替の作業領域として使用 |
| 学習データ書込みコスト | FRAMを第1層キャッシュとしアイドル時にSPI Flashへ非同期書戻し |
| UIが重い | ATtiny1604に全委譲・UIAPduinoはエンジンコアに専念 |

## 4. 実データ解析結果（IPADIC全数解析）

### 4.1 エントリ統計

| 項目 | 数値 |
|---|---|
| 総エントリ数 | 224,150語 |
| 品詞種類数 | 10種類（動詞130,750 / 名詞61,870 / 形容詞27,210 他） |
| 右文脈IDの種類 | 1,290種類 |
| 読みのユニーク数 | 93,372 |
| 最大同音異義語数 | 53語（「カン」「コウ」等） |
| 平均同音異義語数 | 2.4語 |
| 表層形の平均文字数 | 3.7文字 |
| 99%カバーの最大文字数 | 7文字 |

### 4.2 辞書レコード設計（確定版）

PC側コンパイル時に表層形をすべて恵梨沙フォントindexに変換済み。マイコン側の変換コストはゼロ。

```c
typedef struct {
    uint8_t  yomi[8];        // 読み（独自1B/文字エンコード・最大8文字）
    uint16_t elysia_idx[7];  // 恵梨沙フォントindex変換済み（最大7文字×2B）
    uint16_t pos_flags;      // 品詞フラグ（10bit・大分類10種→10bit圧縮）
    int16_t  cost;           // 生起コスト（int16_t）
    uint16_t next_offset;    // 同音異義語次エントリへの相対オフセット
} WnnRecord;  // 計28B
```

### 4.3 辞書全体サイズ

| 項目 | サイズ |
|---|---|
| 辞書本体（224,150エントリ×28B） | 6.00MB |
| 読み2文字インデックステーブル（3,741エントリ×4B） | 14.6KB |
| 合計 | 約6.01MB |
| SPI Flashアプリ領域（15.4MB）への収まり | ✅（残り9.4MB・他アプリ共存可能） |

### 4.4 変換処理速度試算

| 処理 | 時間 |
|---|---|
| 1レコード読み出し（SPI 24MHz） | 約9.5µs |
| 5文節分辞書検索（平均2.4語/読み） | 約114µs |
| Viterbi計算（25経路） | 約2.6µs |
| 合計 | 約0.1ms（ユーザ入力速度の1/1000以下） |

## 5. PC側辞書コンパイラ仕様

### 5.1 変換パイプライン

辞書コンパイルはPC上でPythonにより一度だけ実行する（処理時間：約1〜2分）。

```
IPADIC表層形（EUC-JP）
    → Unicode変換
    → JIS区点（ku, ten）変換
    → 恵梨沙index = (ku-1) × 94 + (ten-1)
    → uint16_t[7] としてWnnRecordのelysia_idxフィールドへ格納
    → SPI Flash用バイナリとして出力（wnn_dict.bin）
```

### 5.2 固定インデックス化

読みの先頭2文字（カタカナ）をキーとして乗算1発でエントリブロック先頭アドレスを計算する。マイコン側での二分探索・ハッシュ計算を完全に排除する。

```c
// マイコン側（たった1行）
uint32_t addr = DICT_BASE + ((yomi_idx) * sizeof(WnnRecord));
```

### 5.3 品詞フラグ化

右文脈ID 1,290種類を品詞大分類10種類にグループ化し、10bitフラグとして各レコードに埋め込む。接続可否の判定はAND演算1クロックで完了する。

```c
// マイコン側（たった1命令）
bool connectable = (prev_word.pos_flags & next_word.pos_flags) != 0;
```

## 6. ATtiny1604 フロントエンド設計

### 6.1 日本語入力フロントエンドの処理分担

| フェーズ | 担当 | UIAPduinoの負荷 |
|---|---|---|
| ひらがな入力 | ATtiny1604完結 | なし |
| 変換要求送信 | ATtiny1604→UIAPduino（I2C） | IAP引数エリア受取のみ |
| 辞書引き・Viterbi | UIAPduino（Wnnブロック） | Wnnコア処理 |
| 候補データ受信 | ATtiny1604←UIAPduino（I2C） | IAP戻り値エリア書込のみ |
| 候補スクロール表示 | ATtiny1604完結（Flashバッファから直接OLED描画） | なし |
| 文節伸縮（左右レバー） | ATtiny1604がコマンドをI2Cで通知・UIAPduinoが再計算 | 再変換のみ |
| 確定 | ATtiny1604がindexをI2Cで返送 | IAP戻り値エリア読取のみ |

## 7. WnnブロックのApp Area設計

### 7.1 サイズ制約と分割戦略

Wnnブロックに割り当てるApp Area枠は最大4KB。理想は1〜2KB。

| 機能 | 推定サイズ |
|---|---|
| 固定インデックス引き当て（乗算+SPI Flash読み出し） | 約40B |
| 品詞フラグAND判定 | 約10B |
| Viterbi簡易版（ビーム幅=1） | 約160B |
| FRAM読み書きラッパー（既存spi_manager流用） | 約80B |
| IAP_ARG/RET_EXTERNAL受け渡し | 約40B |
| 恵梨沙fontindex→I2C送信 | 約30B |
| 合計（Viterbi込み） | 約360B（2KB枠に対し5倍以上の余裕） |

### 7.2 IAPネスト構造

| ケース | Flashロード回数 | 所要時間 |
|---|---|---|
| 単純変換 | 1回（Wnnエントリのみ） | 約110ms（入力中に完了） |
| 複雑変換 | 2回（エントリ＋Viterbi） | 約220ms（入力中に完了） |
| iap_return（アプリ復帰） | 0回（FRAMからSRAM復元のみ） | 約0.7ms |

### 7.3 IAP引数・戻り値エリア（FRAM固定領域）

```c
IAP_ARG_EXTERNAL  0x00016280  // 引数（64B）  呼び出し元→Wnn
                              // 内容: ひらがな文字列（最大32文字・独自1B/文字）

IAP_RET_EXTERNAL  0x000162C0  // 戻り値（64B）Wnn→呼び出し元
                              // 内容: 恵梨沙fontindex列（uint16_t×最大28文字）
```

## 8. Wnn→ATtiny1604 候補データパケット設計

### 8.1 パケットサイズ試算

| 案 | 内容 | サイズ |
|---|---|---|
| 案A（推奨） | 恵梨沙fontindex列（uint16_t）で返す | 25候補×3文字×2B+16B=166B ✅ |
| 案B | 各文節3候補に絞る（フォント生データ） | 5文節×3候補×3文字×8B+16B=376B ✅ |
| 案C | 平均2文字/候補でフォント生データ | 5文節×5候補×2文字×8B+16B=416B ✅ |

案A（fontindexで返す）が最もコンパクトで、ATtiny1604側でのOLED描画に最適。

### 8.2 I2C転送時間

最大376B @ I2C 400kHzで約10ms。ユーザ体感上は問題なし。
